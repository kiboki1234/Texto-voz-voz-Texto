Plan Detallado De Desarrollo — AgroMetrIA
Resumen
Construir AgroMetrIA como una plataforma web agrometeorológica para el reto DevIAthon UPEC. La solución tendrá un backend FastAPI de solo lectura sobre SQL Server Insights y un frontend React + Vite con tres dashboards diferenciados: científico, GAD/institucional y agricultor.

La prioridad técnica es convertir la base YDOC, que usa un modelo vertical tipo locations -> tags -> measurements/recentvalues, en datos claros para visualización, alertas y recomendaciones. La prioridad de producto es maximizar la rúbrica: UI/UX, ejecución técnica, IA/reglas agrometeorológicas, pitch y viabilidad.

Stack decidido:

Backend: Python 3.11+, FastAPI, Pydantic, pyodbc, SQLAlchemy Core opcional, pytest.
Frontend: React + Vite + TypeScript, React Router, TanStack Query, Tailwind CSS, Apache ECharts.
Datos: SQL Server clonado como fuente primaria de solo lectura.
IA/Motor inteligente: reglas agrometeorológicas explicables, no modelo generativo para decisiones críticas.
Deploy demo: local o Docker Compose. Producción futura: backend detrás de API, frontend estático.

Decisiones obligatorias:

No modificar la base YDOC.
No crear relaciones físicas en SQL Server.
No consultar measurements sin filtro de estación, variable y rango de fecha.
Usar recentvalues para tiempo real.
Usar measurements solo para históricos y agregados.
Normalizar HUACA en backend porque su nomenclatura es distinta.
Implementar umbrales propios porque los límites YDOC están en valores genéricos 0/100.
Modelo De Datos Lógico
Relaciones a documentar y usar:

locations.LOCID -> tags.LOCID
params.PARID -> tags.PARID
tags.TAGID -> recentvalues.TAGID
tags.TAGID -> measurements.TAGID
locations.LOCID -> DigitalAssets.LOCID
locations.LOCID -> comments.LOCID
Tablas principales:

locations: estaciones meteorológicas.
tags: sensores/variables por estación.
params: unidades físicas.
recentvalues: última lectura por sensor.
measurements: histórico completo.
DigitalAssets: información RTU.
comments: bitácora.
Normalización de variables:

AVGTA, MINTA, MAXTA: temperatura.
AVGRH, MINRH, MAXRH: humedad relativa.
PR: lluvia.
AVGSR, MINSR, MAXSR: radiación solar.
AVGWSS, MINWSS, MAXWSS: viento sónico.
AVGWSM, MINWSM, MAXWSM: viento mecánico.
AVGWDS, MINWDS, MAXWDS: dirección viento sónico.
AVGWDM, MINWDM, MAXWDM: dirección viento mecánico.
DC: batería.
AVGHH, MINHH, MAXHH, HH: humedad de hoja.
N, P, K: nutrientes del suelo.
Caso HUACA:

Tiene 27 tags, no 29.
Usa nombres antiguos como AvgTemp, MinTemp, HumHoja, Voltaje.
Su NPK usa PARID = 1, no PARID = 15.
El backend debe mapear HUACA a nombres estándar y marcar advertencia técnica en científico.

Reglas Agrometeorológicas
Helada:

Riesgo: Temp_Min <= 2.
Helada inminente: Temp_Min <= 0.
Blanca: Temp_Min <= 2 y Humedad_AVG >= 70.
Negra: Temp_Min <= 2 y Humedad_AVG < 70.
Punto de rocío:

Fórmula Magnus usando temperatura y humedad relativa.
Mostrar en científico y GAD.
Ventana de fumigación:

Apta si:
VV_Sonic_AVG < 3 m/s
Lluvia == 0
10 <= Temp_AVG <= 28
Hum_Hoja_AVG < 70
Si falla, devolver motivos concretos.
NPK:

Nitrógeno:
deficiente < 20
óptimo 20–40
exceso > 40
Fósforo:
deficiente < 10
óptimo 10–30
exceso > 30
Potasio:
deficiente < 80
óptimo 80–200
exceso > 200
HUACA: mostrar advertencia por unidad no confiable.
ET₀:

Usar Hargreaves-Samani en MVP.
Requiere Temp_Min, Temp_Max, Temp_AVG.
Si no hay radiación extraterrestre por latitud/altitud, usar versión simplificada y documentar limitación.
Ombrotérmico:

Agrupar por mes.
Calcular:
T = promedio mensual Temp_AVG
P = suma mensual Lluvia
período seco si P <= 2T.
Batería:

Si unidad real es 4V, umbral operativo: < 3.7V.
Si se confirma sistema 12V, umbral: < 11.5V.
Default MVP: usar umbral por rango observado; documentar la duda.

Riesgos Y Mitigaciones
Riesgo: SQL Server no disponible durante demo.
Mitigación: mocks JSON versionados y modo VITE_USE_MOCKS=true.

Riesgo: consultas históricas lentas.
Mitigación: exigir from/to, usar TAGID, agregar por hora/día para rangos largos.

Riesgo: HUACA rompe comparaciones.
Mitigación: normalizador obligatorio y advertencia visible.

Riesgo: umbrales YDOC no sirven.
Mitigación: motor de reglas propio documentado.

Riesgo: ET₀ cuestionable por datos faltantes.
Mitigación: usar Hargreaves-Samani simplificado y declarar supuestos.

Riesgo: mapa sin coordenadas reales.
Mitigación: usar lista geográfica/diagrama territorial si no hay lat/lon confiables.

Supuestos Y Defaults
Se trabajará sobre SQL Server clonado Insights.
El acceso será de solo lectura.
React + Vite reemplaza Astro por mayor ajuste a dashboards interactivos.
Redis no se incluye en MVP; cache TTL en memoria basta para demo.
No habrá autenticación en MVP público; si se requiere, se añade después.
La IA será explicable mediante reglas agrometeorológicas, no LLM tomando decisiones.
ET₀ será aproximada en MVP.
Las reglas de fumigación y NPK son recomendativas, no prescripción agronómica definitiva.
